# Dillma — Business Requirements

## 1. Product Overview

Dillma is an open-source web service that sends a single prompt to multiple AI models simultaneously. A designated chairman model analyzes all responses and synthesizes a concise, high-quality answer. The service is available as a public instance and for self-hosted deployment.

## 2. Target Audience

General audience — from casual users to developers and researchers.

## 3. Key Use Cases

- Obtaining more accurate and balanced answers through multi-model synthesis.
- Comparing responses from different models to the same prompt.

## 4. Functional Requirements

### 4.1. Providers and Models

- Users connect providers (OpenAI, Anthropic, Google, and other API-compatible services) via their own API keys.
- Keys are stored on the server in encrypted form.
- Users only see models from providers for which they have active keys.

### 4.2. Request Configuration

- Users select 1 to 4 models: up to 3 regular models + 1 chairman.
- If only one model is selected, it acts as the chairman (essentially a standard chat mode).
- Each model is individually configurable: temperature, system prompt, max tokens.
- The chairman's system prompt can include synthesis instructions (e.g., "focus on practicality").

### 4.3. Presets

- Users can save a configuration (model list, their parameters, chairman assignment) as a preset.
- Presets **do not contain** API keys — only model identifiers and their configurations.
- Presets can be published to the catalog.
- If a user lacks keys for some models in a public preset, they cannot use it until the required providers are connected.
- Authors can update published presets. Other users apply updates manually via an "Update" button.

### 4.4. Preset Catalog

- Searchable catalog of published presets.
- Sharing via direct link.
- Rating system based on likes.

### 4.5. Request Processing

- The user's prompt is sent to all selected regular models in parallel.
- Regular models have no chat history. To improve response quality, the system provides them with a brief summary of the current conversation, generated by the chairman.
- The service waits for all model responses. On failure — retry (limited number of attempts). If a model is permanently unavailable, it is excluded from the current request.
- Responses are passed to the chairman **anonymously** (Model #1, Model #2, etc.) — without revealing model names, to eliminate bias.
- The chairman synthesizes the final answer.

### 4.6. Result Display

- The chairman's response is displayed as the primary answer.
- Users can expand and view any individual model's response.
- Users see the mapping: "Model #1 → Claude 3.5 Sonnet", etc.

### 4.7. Revote

- Users can selectively re-run any model, including the chairman.
- When a regular model is re-run, the chairman automatically re-synthesizes the answer.
- On each revote, the anonymous mapping (Model #1, #2...) is randomized anew.

### 4.8. Chat and Sessions

- Multi-turn conversation is supported.
- Only the chairman maintains history: user prompts + chairman responses.
- The model composition can be changed between messages within a session.
- Users can clear a session.
- Sessions persist between visits (chat list).
- Maximum of 5 saved sessions per user.

### 4.9. Authentication and Authorization

- Registration and login via email + password.
- Key storage, sessions, and presets are tied to user accounts.
- Users can delete their account along with all associated data (keys, sessions, presets).

### 4.10. API

- A single API used by both the frontend and external consumers directly.
- The API covers the full functionality of the service.

### 4.11. Rate Limiting

- Mandatory, even when users provide their own keys — to protect the service infrastructure.

## 5. Deployment

- **Public instance** — deployed and maintained by the project team.
- **Self-host** — users deploy the full stack (frontend + backend) on their own infrastructure. The preset catalog is local (not synced with the public instance).
